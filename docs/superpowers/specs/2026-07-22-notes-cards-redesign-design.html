知识碎片页 (NotesCards) 重设计

日期：2026-07-22 作者：sanki 状态：待实现

背景

当前 blogs/notes.md 使用 NotesCards.vue 布局，把 H2 分为主题、H3 是问题、::: details 内是参考答案，前端渲染成两列 CSS grid 卡片。已在生产运行的实现存在以下痛点：

  1. 找不到东西：无搜索、无过滤，14 道题以上就只能滚动扫读；随着题量增长会越发难用。
  2. 审美与站点气质不搭：卡片、圆角、阴影、字体没有明显个性，视觉平淡，缺乏"翻笔记"的手感。
  3. 展开顶起同行邻居：CSS grid 同一 row 高度锁死，任一卡展开会把同行邻居也拉高，视觉跳变。

目标

  • 让"找题"这件事在任何题量下都保持轻松：顶部搜索 + 主题 chip 过滤。
  • 让页面视觉有"温度笔记本"手感（柔和圆角、淡主题色底、主题色染色阴影），并与站点整体气质呼应。
  • 让展开动作只影响自己那一列的后续卡片，不再顶起同行邻居。

非目标

  • 不改动 notes.md 的写法（依旧 H2 主题 / H3 问题 / ::: details 参考答案）。
  • 不新增题目状态区分（有答案 / 待补 / 有来源），空答案统一展示占位文案。
  • 不做多维过滤（难度、标签、掌握度等），只做 A2 级别的"搜索 + 单主题 chip"。
  • 不为搜索命中在答案正文里的场景做特殊 UI（搜索范围只覆盖问题标题）。
  • 不引入吸顶工具栏、抽屉/Modal 展开、URL query 持久化等更重的方案。

用户体验设计

页面结构（从上到下）

┌────────────────────────────────────────────┐
│  🧩 知识碎片                                │  ← 页头（标题 + 一句描述）
│  零散但重要的知识点合集,随手记,持续更新…    │
│                                             │
│  ┌──────────────────────────────────────┐  │  ← 工具栏(平铺,不吸顶)
│  │ 🔎 搜索问题…                          │  │
│  └──────────────────────────────────────┘  │
│  [全部 14] [JavaScript 核心 4] [浏览器 3]… │  ← 主题 chip(单选,带计数)
│                                             │
│  ── JavaScript 核心 ────────────────────    │  ← 分组标题(命中为空时隐藏)
│  ┌──────┐ ┌──────┐                          │
│  │ 卡片 │ │ 卡片 │  ← masonry 瀑布流         │
│  └──────┘ └──────┘                          │
│  ── 浏览器与网络 ─────────────────────       │
│  …                                          │
└────────────────────────────────────────────┘
  • 页头保持当前文案（frontmatter.title || '知识碎片' + 描述），排版细节按段 2 调整字体权重。
  • 工具栏和 chip 不吸顶，平铺在页头下方。
  • 主题分组标题保留（不再删除），在当前过滤下没有命中的分组直接不渲染。

搜索与过滤（A2 / I1 / J1 / K3）

  • 搜索范围：只匹配问题标题（H3 文本），不匹配答案正文（I1）。
  • 匹配方式：大小写不敏感的子串匹配（.toLowerCase().includes()），不做模糊/分词。
  • 搜索与 chip 的交互：
    • 底层过滤规则是 AND：只显示"属于当前 chip 主题（或全部）且 标题命中搜索词"的题。
    • UX 层加一条特例：用户在搜索框键入时，chip 自动被重置为"全部"。理由是"敲搜索"的意图 = "我想扩大范围"；如果不重置，用户在某个 chip 下搜索无结果会困惑于"是搜索没命中还是被 chip 挡住了"。
    • 用户点选任一 chip → 搜索框保持当前内容，过滤在被选主题内生效。
    • 默认状态：chip 选中"全部"，搜索框为空 → 显示所有主题所有题。
  • chip 计数：显示"命中当前搜索条件"的题数。空搜索时就是各主题实际题数；有搜索时会随输入实时更新。计数为 0 的主题 chip 依然显示，但视觉上淡化（禁用态）且不可点。
  • 无结果空态（J1）：
    • 展示搜索图标 + 没找到 "<关键词>" 相关的题 + [清空搜索] 按钮。
    • 点击按钮清空搜索词，回到"全部"状态。
  • 持久化（K3）：
    • 搜索词和 chip 选中状态存 localStorage，键名 notes-cards:filter，格式 { query: string, topic: string }。
    • 页面加载时读取并恢复；用户交互时写入。
    • 不使用 URL query，不做前进后退支持。

卡片布局：Masonry（G2）

  • 用 CSS columns 实现瀑布流，不使用 grid-template-rows: masonry（兼容性问题）。
  • 响应式列数：
    • <640px：1 列
    • 640px – 1200px：2 列
    • ≥1200px：3 列
  • 页面容器 max-width：从当前 1100px 调整到 1280px，为 3 列布局留出舒适的卡片宽度。
  • 卡片 break-inside: avoid，用 margin-bottom 而不是 gap 控制间距（columns 布局特性）。
  • 卡片间距（垂直）：20px。
  • 同一列内展开：只顶起同列后续卡片，不影响其他列。这是 masonry 的天然属性，正是本次改造要解决 G 问题的核心手段。

卡片视觉（F2 温度笔记本）

卡片折叠态：

┌───────────────────────────────────────┐
│▍ 什么是 Promise?如何手动实现…    ▾    │
└───────────────────────────────────────┘

卡片展开态：

┌───────────────────────────────────────┐
│▍ 什么是 Promise?如何手动实现…    ▴    │
│                                       │
│  参考答案内容,支持 markdown 渲染,     │
│  代码块、段落、列表都会正常显示。      │
│  ┌─────────────────────────────────┐  │
│  │ const promise = new Promise(...) │  │
│  └─────────────────────────────────┘  │
└───────────────────────────────────────┘

关键视觉规则：

  • 主题色映射（3 个主题固定映射，写成常量表）：
    • JavaScript 核心 → #f5a623（温暖黄橙）
    • 浏览器与网络 → #4a90e2（标准蓝）
    • 框架与工程化 → #7c3aed（紫）
    • 主题名未匹配时 fallback 到中性灰 #6b7280。
  • 卡片底色：淡主题色浅底 rgba(主题色, 0.03)（浅色模式）；深色模式下用主题色 0.04 覆盖在深底之上。
  • 展开态底色：0.05 覆盖率（比折叠态略深一档），暗示"这张被翻开了"。
  • 左竖条：宽度 5px（当前 4px），实心主题色；顶部微圆角 2px 呼应笔记本装订线。
  • 卡片圆角：18px（当前 16px）。
  • 卡片边框：保留 1px 淡边框，颜色和当前一致 (rgba(125,125,125,0.15) / 深色 rgba(255,255,255,0.1))。
  • hover 阴影：主题色染色阴影 0 6px 24px rgba(主题色, 0.12)，替代当前的黑色阴影。深色模式下 0.20。
  • hover transform：保留 translateY(-2px)，过渡曲线改为 cubic-bezier(0.4, 0, 0.2, 1)。
  • 问题标题字体：font-weight: 650（当前 600），字体继承 UI 字体栈（不切换成得意黑，避免过重）。
  • 展开箭头：从 CSS border 拼的 ▾ 改成内联 SVG chevron，展开态旋转 180°；过渡曲线同上，时长 350ms。
  • 卡片内 padding：折叠态 14px 18px；展开态答案区 0 18px 18px（和标题左右对齐）。

分组标题（H2 主题）：

  • 主题色小圆点（8px 圆）+ 主题名（font-weight: 700, 得意黑字体）+ 题数文字（"N 题"，灰度 60% 主题色）
  • 下方一条 2px 主题色渐变分割线（linear-gradient(to right, 主题色 20%, transparent)）。
  • 分组之间的垂直间距 48px（当前 3rem = 48px 保持不变）。

展开交互（保留高度动画）

  • 保留当前 grid-template-rows: 0fr → 1fr 的展开高度动画（原生 CSS 无 JS 高度动画）。
  • 过渡时长 350ms（当前 300ms），曲线 cubic-bezier(0.4, 0, 0.2, 1)。
  • 多张同时展开：允许，每张卡独立状态，masonry 天然支持。
  • 展开状态不持久化：每次进入页面所有卡都是折叠态。搜索/过滤状态才是全局意图，值得持久化；展开是单卡片的一次性行为。

答案区渲染细节

  • Markdown 已在 config.js 的 onInitialized 钩子里预渲染成 HTML 注入 frontmatter.__cards，本次改造不动这块。
  • 代码块横向滚动：masonry 列宽变窄后，答案里的代码块 (.notes-card__a-content pre) 必须能横向滚动，加 overflow-x: auto，不撑破卡片。
  • 深色模式代码块背景：需要和卡片底色区分（当前的主题继承色可能与浅主题色底冲突），代码块背景显式设为 rgba(0,0,0,0.35) 或类似深灰。
  • 空答案（H3）：当 item.a 为空字符串或只有空白时，答案区渲染占位文案"参考答案待补",样式：斜体 + 主题色 60% 灰度。这个判断逻辑在 NotesCards.vue 里做（模板层三元 or 计算属性）。

技术设计

组件与文件

改动收敛在两处，不新增文件：

  • .vuepress/layouts/NotesCards.vue
    • 新增顶部工具栏（搜索框 + chip 组组件）。
    • 新增 <script setup> 内的状态：searchQuery、selectedTopic（值为主题字符串或 '__all__'）。
    • 新增计算属性 filteredCards：基于 cards、searchQuery、selectedTopic 派生。
    • 新增 localStorage 读写（onMounted 读、watch 双状态写）。
    • 空答案分支渲染占位文案。
  • .vuepress/style/notes-cards.scss
    • 页面 max-width 改为 1280px。
    • .notes-grid 改用 CSS columns。
    • 新增工具栏、搜索框、chip 组、空态样式。
    • 卡片视觉按 F2 规则更新（底色、圆角、竖条、hover 阴影、字体权重等）。
    • 主题色映射写成 SCSS map，模板层通过 :style 注入。

主题色映射（常量）

在 NotesCards.vue 顶层维护，替代当前 TOPIC_COLORS 数组：

const TOPIC_COLOR_MAP: Record<string, string> = {
  'JavaScript 核心': '#f5a623',
  '浏览器与网络': '#4a90e2',
  '框架与工程化': '#7c3aed',
};
const FALLBACK_COLOR = '#6b7280';
const topicColor = (topic: string) => TOPIC_COLOR_MAP[topic] ?? FALLBACK_COLOR;

新增主题时，手动在这里加一条映射即可。

状态与过滤逻辑

// 状态
const searchQuery = ref('');
const selectedTopic = ref<string>('__all__'); // '__all__' | 主题名

// 派生:命中的题目按主题重新分组
const filteredCards = computed(() => {
  const q = searchQuery.value.trim().toLowerCase();
  return cards
    .map((topic) => {
      const inTopic = selectedTopic.value === '__all__' || selectedTopic.value === topic.topic;
      if (!inTopic) return { ...topic, questions: [] };
      const matched = topic.questions.filter((item) =>
        q === '' || item.q.toLowerCase().includes(q)
      );
      return { ...topic, questions: matched };
    })
    .filter((topic) => topic.questions.length > 0);
});

// chip 计数(所有主题,包括命中为 0 的)
const topicCounts = computed(() => {
  const q = searchQuery.value.trim().toLowerCase();
  const all = cards.reduce((sum, t) =>
    sum + t.questions.filter(x => q === '' || x.q.toLowerCase().includes(q)).length, 0);
  return {
    __all__: all,
    ...Object.fromEntries(cards.map(t => [
      t.topic,
      t.questions.filter(x => q === '' || x.q.toLowerCase().includes(q)).length
    ]))
  };
});

// 键入搜索时清空 chip
watch(searchQuery, () => { selectedTopic.value = '__all__'; });

localStorage 持久化

  • 键名：notes-cards:filter
  • 结构：{ query: string, topic: string }
  • onMounted 读一次并回填状态；两个 watch 分别监听 searchQuery 和 selectedTopic，任一变更就写入。
  • SSR 场景要 guard typeof window !== 'undefined'，避免构建期报错。

空态

一个简单的独立块，条件是 filteredCards.value.length === 0 && (searchQuery.value !== '' || selectedTopic.value !== '__all__')：

  • 搜索图标（内联 SVG 或字体 icon）
  • 文案：没找到 "<关键词>" 相关的题（有关键词时展示关键词；无关键词但 chip 选了某主题时展示"该主题下没有题"）
  • [清空搜索] 按钮 → 重置 searchQuery 和 selectedTopic

兼容性与降级

  • CSS columns 兼容性覆盖所有目标浏览器（IE 早已不支持）。
  • masonry 里代码块横向滚动 overflow-x: auto 是原生行为，无需额外 JS。
  • 深色模式所有颜色调整都在 html.dark .notes-card 选择器分支下。
  • 无 localStorage 环境（禁用/隐私模式）时 try/catch 静默降级，功能可用只是不记忆状态。

边界

  • 主题色映射里没有的主题（未来新增）会 fallback 到中性灰，视觉上不会崩，但会失去"温度笔记本"的手账感 → 提醒未来加主题时同步加色。
  • 搜索命中在答案里的场景不支持（用户看到"命中数为 0"可能困惑于"我明明在某题答案里见过这个词"）→ 由 I1 决策明确排除。
  • 空答案卡片高度会比有答案的短，masonry 下自然错落，不需要特殊处理。

测试计划

只做本地手工验证，不加自动化测试：

  • 搜索：
    • 空搜索：显示所有题，chip 计数 = 各主题题数
    • 大小写不敏感命中：搜"promise"、"PROMISE" 都命中同一题
    • 无结果：正确显示 J1 空态和关键词
    • 键入搜索时 chip 自动回到"全部"
  • 主题 chip：
    • 单选切换正确
    • 计数随搜索实时更新
    • 计数为 0 的 chip 视觉淡化且不可点
  • 持久化：
    • 输入搜索、切 chip、刷新页面 → 状态恢复
    • 清空搜索按钮 → 同步清空 localStorage
  • 展开：
    • 单卡片展开/折叠动画流畅
    • 多卡片同时展开
    • 展开只顶起同列后续卡片，不影响其他列（验证 G2 落地效果）
    • 空答案卡展开显示占位文案
  • 视觉：
    • 三个主题色底色可辨（非纯白）
    • hover 阴影是主题色染色
    • 圆角、竖条、字体权重达预期
    • 深色模式各项对比度足够
  • 响应式：
    • <640px 单列
    • 640-1200px 双列
    • ≥1200px 三列
    • 移动端搜索框和 chip 不溢出
